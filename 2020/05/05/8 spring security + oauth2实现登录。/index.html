首先需要测试获取token（即登录接口），获取token的路径是 /oauth/token，请求是post请求，参数有三个一个是用户名username，一个是登录的密码password，还有一个是grant_type，这个参数表示使用的oauth2模式，前面说过了，一共有四种，我们本次使用的是密码模式，所以它的值为password。当然有个三个参数还不够，还需要在请求头中，传入一个Authorization 参数，这个参数需要传之前在认证服务器（AuthorizationServerConfig）中配置的客户端id和密码的base64的编码。（如果没有在请求的header携带Authorization，会报401）
在本次的测试中使用的clientId为security，密码为123456，base64编码后为：c2VjdXJpdHk6MTIzNDU2。注意Authorization值的格式为Basic c2VjdXJpdHk6MTIzNDU2
这个编码随便百度一个base64在线加密的网站就加密就行。

登录成功后，服务器会返回这个access_token，你如果向访问受保护的资源，就需要在请求的url后边携带这个access_token，服务器才知道你登录成功的用户，可以识别你的身份。

到这里基本的测试接口就结束了。下面是配合前后端分离登录的问题及解决办法。

这里我先解释下401和403吧，因为前后端分离稍不留神就报这两个状态码。

401: 未授权（Unauthorized）。有两种情况会发生这个问题。

1: 当你访问 /oauth/** 路径时，忘记了在header里加Authorization 参数。
2: 当你为登录而访问受保护的资源时。（一般就是访问自己的Controller）。

403: 拒绝访问（Forbidden ）可以简单的理解为没有权限访问。这个问题一般是后端设置了访问权限。

cors和csrf

cors: Cross Origin Resourse-Sharing - 跨站资源共享，即我们熟知的跨域问题。相信很多同学都知道怎么解决前后端分离的跨域问题。
csrf: Cross-Site Request Forgery - 跨站请求伪造，是一种网络攻击方式，该攻击可以在受害者毫不知情的情况下以受害者名义伪造请求发送给受攻击站点，从而在未授权的情况下执行在权限保护之下的操作，具有很大的危害性。
